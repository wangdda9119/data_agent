access 토큰 블랙리스트: bl:<access_jti> → TTL=access 만료까지

“redis 기반 jwt관리”를 가장 깔끔하게 만족시키는 방식은 refresh를 서버(=Redis)에서 통제하고, access는 짧게 가져가며 필요 시 블랙리스트를 쓰는 방식임.

# 실행계획.md

## 0) 목표 / 범위

- 프론트: Vue (변경 최소, 백엔드 API 연동만)
- 백엔드: Python (기존 프로젝트에 DB/인증 기능 추가)
- DB 설정: 현재 없음 → **PostgreSQL + Redis를 Docker로 구축**
- 테이블: **user 테이블 1개만 생성** (추가 테이블 금지)
- 인증/인가: **JWT 기반**, Redis로 토큰 상태 관리(로그아웃/재발급 통제)
- 구현 제약: **파이썬 라이브러리만 사용** (DB/Redis/JWT/웹프레임워크 포함)

---

## 1) 전체 구조 및 인증 흐름

### 1.1 구성 요소
- `backend` (Python API)
- `postgres` (user 저장)
- `redis` (JWT 상태 관리)
- `frontend` (Vue)

### 1.2 권장 인증 모델 (Access + Refresh)
- Access Token: 짧은 만료(예: 15분)
- Refresh Token: 긴 만료(예: 14일)
- Redis 사용 목적
  - Refresh 토큰 **화이트리스트(허용 목록)** 저장 → 재발급 통제
  - Access 토큰 **블랙리스트(로그아웃/강제 만료)** 저장(선택/권장)

### 1.3 토큰 키 설계 (Redis)
- Refresh 화이트리스트
  - Key: `rt:{user_id}:{refresh_jti}`
  - Value: `"1"` 또는 JSON(필요 시)
  - TTL: refresh 만료시간과 동일
- Access 블랙리스트
  - Key: `bl:{access_jti}`
  - Value: `"1"`
  - TTL: access 토큰의 남은 만료시간

## 2) DB 설계 (user 테이블 단 1개)

### 2.1 user 테이블 컬럼 (필수 최소)
- `id` BIGSERIAL PRIMARY KEY
- `email` VARCHAR(255) NOT NULL UNIQUE
- `password_hash` TEXT NOT NULL
- `nickname` VARCHAR(100) NULL (또는 NOT NULL 정책 선택)
- `is_active` BOOLEAN NOT NULL DEFAULT TRUE
- `created_at` TIMESTAMPTZ NOT NULL DEFAULT NOW()
- `updated_at` TIMESTAMPTZ NOT NULL DEFAULT NOW()

### 2.2 DDL (PostgreSQL)
```sql
CREATE TABLE IF NOT EXISTS "user" (
  id BIGSERIAL PRIMARY KEY,
  email VARCHAR(255) NOT NULL UNIQUE,
  password_hash TEXT NOT NULL,
  nickname VARCHAR(100),
  is_active BOOLEAN NOT NULL DEFAULT TRUE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- updated_at 자동 갱신을 애플리케이션 레벨에서 처리 (테이블 추가 금지 조건 준수)

2.3 비밀번호 정책

저장: 평문 금지, bcrypt/argon2 기반 해시

검증: 해시 검증 함수로 비교

파이썬 라이브러리: passlib[bcrypt] 또는 bcrypt 단독 사용

4.3 엔드포인트 정의 (최소)

POST /auth/signup

Input: email, password, nickname(선택)

Flow:

email 중복 검사

password 해시 생성

user 저장

응답(민감정보 제외)

POST /auth/login

Input: email, password

Flow:

user 조회 + is_active 확인

password 검증

access_jti/refresh_jti 생성

access_token/refresh_token 발급

Redis에 refresh 화이트리스트 저장(rt:{user_id}:{refresh_jti} TTL=refresh)

응답: tokens

POST /auth/refresh

Input: refresh_token

Flow:

refresh_token 검증(서명/만료)

(핵심) Redis 화이트리스트 존재 확인(rt:{user_id}:{refresh_jti})

새 access_token 발급(+선택: refresh rotation)

응답: 새 access_token (및 rotation 시 새 refresh_token)

POST /auth/logout

Header: Authorization: Bearer access_token

Body(optional): refresh_token

Flow:

access_token에서 jti 추출 → Redis에 블랙리스트 등록(bl:{access_jti} TTL=access 남은 시간)

refresh_token을 받았다면 refresh_jti 추출 → Redis 화이트리스트 삭제(rt:{user_id}:{refresh_jti})

응답: ok

GET /me (인가 확인용 최소)

Header: Authorization

Flow:

access_token 검증

Redis 블랙리스트 체크

user_id로 user 조회 후 반환
